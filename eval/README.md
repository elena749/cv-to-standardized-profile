# Eval Set v1 — CV-to-Standardized-Profile

20 synthetic CVs for evaluating the Build 2 pipeline. All names, employers, and personal details are fictional. CVs follow plausible distribution patterns for a strategy consulting recruiting funnel in DACH.

## Distribution

**Realistic split (Option 1, 70/20/10):**
- 14 standard CVs (typical applicants, plausible career paths)
- 4 edge cases (career switcher, gaps, multi-sector, international)
- 2 breaking cases (very short, very long)

**Language split:**
- 14 German
- 6 English

**Career level split:**
- 6 Entry (Hochschulabsolventen, 0-2 years)
- 8 Mid (Associate, 2-4 years)
- 6 Senior (Senior Associate / Manager, 5-8 years)

**Format split:**
- 8 DOCX (1 complex with tables and multi-column: cv_13)
- 10 PDF (4 simple pandoc-generated, 4 complex layouts, 2 with tables)
- 1 TXT (cv_19)
- 1 PDF (long, complex: cv_20)

**Complex-layout CVs (deliberately included to stress-test parsing):**
- cv_03: 2-column PDF with dark sidebar (DE, Senior)
- cv_07: 2-column minimal PDF with skills sidebar (EN, Mid)
- cv_13: complex DOCX with multi-column header, embedded tables, styled headings (DE, Senior)
- cv_14: modern PDF with colored gradient header banner and pill-style skill tags (EN, Senior)
- cv_17: dense multi-sector PDF with project-card grid layout (EN, Mid)

## What this set tests

**For pipeline correctness:**
- Multiple input formats (DOCX, PDF text, PDF with tables, TXT)
- Multiple languages (DE, EN)
- Multiple career levels
- Schema enforcement (all CVs should produce schema-valid profiles)
- Taxonomy mapping (all extracted values should map to canonical entries or be marked unmapped)

**For failure-mode discovery:**
- cv_15 — career switch from law: tests how non-consulting backgrounds map to functional_expertise
- cv_16 — gaps in employment: tests handling of non-employment periods
- cv_17 — multi-sector: tests whether pipeline preserves all industries or collapses them
- cv_18 — international with non-DACH education: tests education_credentials enum
- cv_19 — minimal data: tests pipeline behavior when most fields cannot be extracted
- cv_20 — long CV with dense data: tests context window limits and extraction precision

## Ground Truth Files

Each CV has a corresponding `ground_truth/cv_XX.json` file. These are the hand-labeled expected outputs against which pipeline runs are scored.

**Status:** Ground truth files are NOT generated by Claude. They must be hand-labeled by the human evaluator (Elena) to ensure honest evaluation. A template is provided in `ground_truth/_template.json`.

## Layout Complexity

This eval set includes 5 CVs with complex layouts (sidebar columns, gradient headers, embedded tables, project-card grids) that stress-test PDF/DOCX parsing beyond pandoc-generated standard structures. The remaining 15 CVs use simpler pandoc-generated layouts.

This is a deliberate trade-off:
- **Strength:** Mix of simple and complex layouts gives separate signals for *"does the LLM extract correctly"* vs *"does the parser handle the layout"*
- **Limitation:** Real-world CVs include even more variation (scanned PDFs, photo-driven Canva templates, multi-language mixed scripts). These are deliberately out of scope for v1 and will be added in the BREAK-Session test set.

When reporting eval metrics, separate F1 by simple vs complex layout to identify whether failures stem from extraction quality or layout parsing.
