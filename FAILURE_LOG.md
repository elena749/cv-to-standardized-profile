# Failure Log

This log documents what broke during development, why, and how it generalizes. Each entry follows: **Timestamp / What I tried / What broke / Root cause / Fix or mitigation / Generalizes to**.

The "generalizes to" line is the most valuable — it's the pattern that informs future builds.

---

## 2026-05-07 — Eval Set Generation: Six Failure Classes Identified

### What I tried
Generated 20 synthetic CVs across DOCX/PDF/TXT formats, then AI-pre-labeled ground truth JSONs using Claude. Hand-validated all 20 ground truth files against source CVs.

### What broke
Six recurring failure classes surfaced during human review of AI-pre-labeled ground truth:

#### 1. Liberal Functional Expertise interpretation (cv_15)
LLM derived Functional Expertise from related terms ("regulated takeovers" → incorrectly mapped to "Risk Management") and from MBA specializations ("Strategy" in MBA → mapped as Functional Expertise).

**Generalizes to:** Pipeline LLM is likely to make the same inference error. Mitigation: stricter prompts ("Extract only operationally performed expertise, not derived from education or context"), or self-critique loop in v2.

#### 2. LLM date arithmetic unreliable (cv_12)
Total_years_experience calculated as 4 instead of 5 (5 years from August 2021 to May 2026 with mathematical rounding).

**Generalizes to:** Years-of-experience and career_level should be computed deterministically in code from extracted employment_history dates, NOT generated by LLM directly. v2-roadmap candidate.

#### 3. Temporal project-to-role attribution (cv_11, cv_12)
When CV separates roles (Position 1, Position 2) from project highlights (Project A in 2024, Project B in 2022), LLM tends to attribute all projects to the most recent role rather than allocating each project by date.

**Generalizes to:** Project-to-role mapping requires temporal logic over CV structure, not just keyword extraction. v2-roadmap: dedicated post-processing step that maps project dates to role periods.

#### 4. Pipeline-input must equal canonical source (cv_13)
Custom DOCX generation script (python-docx) omitted methods, certifications, and dissertation details that existed in the markdown source. Ground truth labeled against the source would have penalized the pipeline for content lost in upstream conversion.

**Generalizes to:** Ground truth must match what the pipeline actually receives, not what was nominally generated. When CV files are generated from a canonical source, the rendered files must be verified to preserve all source content before ground truth labeling.

#### 5. Career level requires title + years, not years alone (cv_13, cv_14)
Original convention: "8+ years = out-of-scope (Lead)". Reality: Senior Manager titles often have 10+ years and remain Senior-track, not Lead/Partner.

**Generalizes to:** Career_level inference should combine title parsing + years math, not just years from total_years_experience.

#### 6. Synthetic CV data must respect firm-specific nomenclature (cv_20)
Original cv_20 used "Senior Manager" at McKinsey (Bain/Roland-Berger nomenclature, not McKinsey's). McKinsey uses "Engagement Manager".

**Generalizes to:** When synthesizing eval data, firm-specific nomenclature matters. Future CV synthesis should use a firm-specific title dictionary.

### Root cause
AI-pre-labeling tends systematically toward liberal interpretation: "sounds related" becomes "is mapped". Without strict definitional discipline, the LLM bridges plausible-but-incorrect inferences.

### Fix
Hand-validation of all 20 ground truth files. Updated `LABELING_CONVENTIONS.md` with 17 conventions, including 8 new ones from this review. All 20 ground truth JSONs schema-validated.

### Generalizes to (most important)
**AI-pre-labeling of ground truth is a valid technique, but only if a human verifies — not durchklickt.** The cost of skipping verification: ground truth itself becomes unreliable, eval results become circular, downstream comparison loses meaning.
